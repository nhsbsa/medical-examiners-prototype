Sprint 18 - ME 60 min timeout
60 min timeout

Hypothesis A

We believe Adding a timeout warning is an essential feature for users, as it will allow them to extend their session if they need to and be aware of the time limit users as this will allow us to comply with data security requirements by having a 60 minute timeout

What we added:

  • Added 3 pages: Timeout, Timing out and Timed out
  • Timeout warning is shown to users 5 minutes before the time runs out
  • Users can choose to click the link to stay signed in or to be signed out

What we plan to test:

  • If users are able to complete the task in the given timeframe
  • If users are aware of timeframe on the page
  • If the warning window is obvious enough
  • If users are concerned about loosing the inputted data

What we found:

  • Users found instructions how to stay logged in clear
  • The pop up warning window makes sense to them
  • They like the fact that it tells them that information won’t be saved

Any known issues?

  • Progressive enhancement
  • If an user don't use Java Script then we need to use meta refresh tag inside Noscript tags that matches the session timeout time
  • If am user is not able to see the warning if away from the screen, it will log them anyway

Sprint 14 &15 - Coroner Notification
Coroner Notification - Iteration 2

Hypothesis A

We believe adding a Coroner Notification section will be a useful feature for users as this will ensure users record information that can be transferred to the Coroner. Users need to fill this section if they are not filling up the Coroner's web portal. This will enable a case to be referred to a coroner so that further independent scrutiny can take place.

What we changed:

  • Added extra screens and changed wording, to help the user understand the options for coroner notification

  • Do users understand what an external referral is, and what they need to do when they complete the referral using the ME service
  • Whether users understand that some information has been filled in other sections within MES will be pull through to this section
  • ..

Coroner Referral - Iteration 1

Hypothesis A

We believe adding a Coroner Referral section will be a useful feature for users as this will ensure users record information that can be transferred to the Coroner. Users need to fill this section if they are not filling up the Coroner's web portal. This will enable a case to be referred to a coroner so that further independent scrutiny can take place.

What we changed:

  • Information will be derived from several parts of the system to be transferred to the Coroner (so that we are not asking the same question twice)
  • There are two types of scenario : Scenario 1 - this is the best case scenario & Scenario 2 - worst case scenario
  • Scenario 1 - where all the fields in the other sections (incl the optional ones) are filled up
  • Scenario 2 - where none of the optional fields been filled

What we plan to test:

  • Are users able to follow the user journey to complete the referral
  • Whether users understand that some information has been filled in other sections within MES will be pull through to this section
  • ..

What we found:

  • Users never had to provide Communicable infections to the Coroner - although this is something that the Stakeholder requested; Following UR - we decided to remove this question
  • Medical Device : users were confused with this term; further discussion within the team we found that this ques is for the FD therefore removed it and will be added to the MCCD service
  • UR indicated cases referred to the Coroner are usually made by the MEO (with the advice from ME); Users were inputting the MEO's name in Reporting person fields - this was changed to Referring doctor as Coroner wants details of someone with the GMC number and not the MEO
  • Further research finding can be found here

Any known issues?

  • No validation errors

MFA - Iteration 1

Hypothesis A

We believe adding a MFA will be a useful feature for users as it is a requirement in terms of security compliant and it enables stronger authentication. This will reduce the risk of security threat.

Took learning from MOT Testing's MFA user journey.

What we changed:

  • Added few screens to the journey to take users through on how to set the authentication app login

What we plan to test:

  • Are users able to follow the steps to fully set the authentication app
  • ..

What we found:

  • Participants understood the need of MFA and was ok to download the app
  • However, participants found difficulities in navigating 3rd party website (Authy) once they have downloaded the app
  • Participant was unsure on how to progress once they have downloaded the app back to our service
  • Participant did not see the key to be entered and went straight to the 2nd point on entering the code & got confused on what to do and did not know how to get the code
  • Participant got confused between the smartphone and desktop app download journey

Any known issues?

  • No validation errors

Case concerns

Hypothesis A

We believe adding a case concern will be a useful feature for users as it will help users to add concerns to a case & to be able to produce report for the quarterly report. This will help users to complete their tasks.

What we changed:
